# 11. 本地測試與除錯規範 (Local Testing & Debugging Strategy)

## 核心守則 (Core Rule)
**「所有程式碼異動與 Bug 修復，必須先在本地端（Localhost）驗證無誤，並經由使用者確認需求達成後，才可詢問使用者是否推送（Push）至 GitHub。」**

## 為什麼需要這樣做？
由於本專案高度依賴 GitHub Actions 進行自動化爬蟲與 GitHub Pages 靜態部署，直接在 `main` 分支進行盲測（Blind Test）會導致以下問題：
1. **觸發無效的 CI/CD**：頻繁的盲測提交會導致排程被無效觸發，浪費額度。
2. **部署時間差**：GitHub Pages 部署通常需要 1~2 分鐘，依賴部署後驗證效率極低，且容易與瀏覽器快取（Cache）混淆。
3. **破壞歷史紀錄**：未經測試的程式碼推送到線上，會污染 Git 紀錄與 `state.json` 的穩定性。

## 標準除錯與測試流程 (Standard Debugging Flow)

### 1. 後端與爬蟲 (Scraper / Notifier)
遇到爬蟲失效或 Telegram 推播失敗時：
*   **拒絕盲推**：嚴禁直接修改原始碼並推送到 GitHub 看是否成功。
*   **本地重現**：使用建立好的本地測試腳本（如 `node test-telegram.js` 或 `node debug-telegram.js`）重現錯誤。
*   **觀察日誌**：在 Console 觀察詳細的 API 錯誤代碼（如 `400 Bad Request`）與資料結構。
*   **驗證修復與確認**：在本地端修正邏輯，確保腳本回傳 `200 OK` 且資料無誤。接著向使用者展示結果，待確認後才可 Commit/Push。

### 2. 前端介面 (Dashboard)
遇到 React UI 跑版、狀態未更新、按鈕殘留時：
*   **啟動本地伺服器**：在 `dashboard/` 目錄下執行 `npm run dev`。
*   **視覺驗證**：打開瀏覽器訪問 `http://localhost:5173/`，確保所做的 UI 修改（例如：移除 DevTools 面板、調整預設 Tab）確實生效。
*   **清理快取**：測試前確保清除 LocalStorage（例如 `etf_visible_list` 或 `etf_mock_user`）以模擬真實使用者的首次載入狀態。

## 防錯機制 (Fail Fast)
AI 在協助修改程式碼時，必須主動遵守此技能：
1. **修改前**：釐清當前問題，並提出本地端測試方案。
2. **修改後**：主動啟動 `browser_subagent` 或 `run_command` 進行驗證，並在回覆中提供本地端的測試日誌或截圖結果。
3. **推播前**：只有在本地測試完全成功，且向使用者展示結果並獲得「需求無誤」的確認後，才可詢問是否執行 `git push origin main`。
